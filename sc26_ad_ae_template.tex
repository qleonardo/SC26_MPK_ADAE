\section{Overview of Contributions and Artifacts}

\subsection{Paper's Main Contributions}

\begin{description}
\item[$C_1$] We present FastMPK, a high-performance sparse matrix power kernel designed to run efficiently on both CPUs and GPUs.

\item[$C_2$] We propose a cache-reuse strategy and develop a dependency-aware parallelization scheme for FastMPK. Our evaluations show significant advantages of FastMPK over state-of-the-art approaches on both CPUs and GPUs.

\item[$C_3$] We demonstrate the practical impact of FastMPK by integrating it into a polynomial preconditioned GMRES solver, where it achieves end-to-end speedups with minimal preprocessing overhead.
\end{description}

\section{Artifact Identification}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\newartifact
\artrel

$A_1$ (FastMPK) is the core computational artifact. It implements multiple MPK variants:
\begin{itemize}
  \item \textbf{CSR-based MPK}: a baseline using standard CSR SpMV.
  \item \textbf{Library-based MPK}: using Intel oneMKL (CPU) or cuSPARSE (GPU) SpMV.
  \item \textbf{FastMPK (w/o Level)}: FastMPK without the level-based traversal optimization.
  \item \textbf{FastMPK}: the full version with cache-reuse blocking, dependency-aware partitioning, and level-based traversal.
\end{itemize}
The CPU version runs on multi-core CPUs (Intel/AMD/ARM); the GPU version runs on NVIDIA GPUs. Both support double-precision computation.

\artexp

The measured performance is reported in GFLOPS. Compared to the baseline CSR-based MPK, library-based MPK, and FastMPK without level optimization, the full FastMPK should show the highest performance on most matrices, demonstrating the effectiveness of the cache-reuse strategy and dependency-aware parallelization ($C_1$, $C_2$).

\arttime

The expected computational times of this artifact are:

\begin{center}
\begin{tabular}{ll}
\toprule
Platform & Estimated Time (mins) \\
\midrule
Intel Xeon Max 9468 (CPU)  & $\sim$180 \\
AMD EPYC 7742 (CPU)        & $\sim$210 \\
Huawei Kunpeng 920 (CPU)   & $\sim$240 \\
NVIDIA H100 (GPU)          & $\sim$120 \\
NVIDIA A100 (GPU)          & $\sim$180 \\
\bottomrule
\end{tabular}
\end{center}

\artin

\artinpart{Hardware}

\emph{CPU platforms}: Intel Xeon Max 9468 @ 2.1\,GHz (x86, 48 cores), AMD EPYC 7742 @ 1.5\,GHz (x86, 64 cores), and Huawei Kunpeng 920 @ 2.6\,GHz (ARM, 32 cores).

\emph{GPU platforms}: NVIDIA H100 (80\,GB) and NVIDIA A100 (80\,GB).

\artinpart{Software}

\emph{CPU}: Intel oneAPI 2025.1 with oneMKL (Intel platform); GCC 11.2.1 (AMD and Huawei platforms).

\emph{GPU}: CUDA 12.1 (H100) or CUDA 11.7 (A100), with cuSPARSE.

\artinpart{Datasets / Inputs}

110 large sparse matrices (with number of non-zeros $\ge$ 8,000,000) from the SuiteSparse Matrix Collection\footnote{https://sparse.tamu.edu/}. An input vector is randomly generated for each matrix. The matrix power $K$ is set to values in $\{1, 3, 5, 7, 9\}$.

\artinpart{Installation and Deployment}

\emph{CPU}: All artifacts are compiled in double precision with \texttt{-O3} and OpenMP enabled. On Intel CPUs we use \texttt{icpc} (Intel oneAPI 2025.1) with MKL; on AMD and Huawei CPUs we use \texttt{g++} 11.2.1. Thread affinity is set to bind each software thread to an individual physical core (no hyperthreading). The number of threads matches the core count of each platform.

\emph{GPU}: Compiled with \texttt{nvcc} using \texttt{-O3 -arch=sm\_80} (A100) or \texttt{sm\_90} (H100) in double precision.

\artcomp

\emph{Workflow:}
\begin{enumerate}
  \item For each of the 110 matrices, generate a random input vector.
  \item Run each MPK variant (CSR / MKL-cuSPARSE / FastMPK-wo-Level / FastMPK) 100 times per matrix per $K$ value.
  \item Record the average execution time and compute GFLOPS as $2 \times \text{nnz} \times K \;/\; (\text{time in seconds} \times 10^9)$.
\end{enumerate}

The CPU version's \texttt{Makefile} produces a single \texttt{main} executable that runs all variants. The GPU version's \texttt{Makefile} produces \texttt{main-double} (and optionally \texttt{main-float}).

\artout

Based on the measured average GFLOPS, we can generate Table IV and plot Figures 3, 5, 9--13. The output CSV file (\texttt{performance\_result.csv}) contains the matrix name, dimensions, and GFLOPS for each variant.


\newartifact

\artrel

$A_2$ integrates FastMPK as a polynomial preconditioner into the GMRES solver from the Ginkgo library\footnote{https://github.com/ginkgo-project/ginkgo}. It compares three GMRES configurations:
\begin{itemize}
  \item \textbf{Ginkgo-native}: GMRES without a preconditioner (baseline).
  \item \textbf{CSR-MPK}: GMRES preconditioned by a CSR-based MPK polynomial (CPU only).
  \item \textbf{FastMPK}: GMRES preconditioned by FastMPK with cache-reuse blocking.
\end{itemize}
The CPU version uses Ginkgo's \texttt{OmpExecutor}; the GPU version uses \texttt{CudaExecutor}. This artifact demonstrates the end-to-end benefit of FastMPK in a real iterative solver ($C_3$).

\artexp

The GMRES solver using FastMPK as the polynomial preconditioner should converge in fewer iterations and achieve lower total execution time compared to the baseline Ginkgo-native GMRES and the CSR-MPK-preconditioned variant, on most of the test matrices. The results demonstrate that FastMPK's preprocessing overhead is amortized by faster convergence.

\arttime

The expected computational times of this artifact are:

\begin{center}
\begin{tabular}{ll}
\toprule
Platform & Estimated Time (mins) \\
\midrule
Intel Xeon Gold 6258R (CPU) & $\sim$20 \\
NVIDIA H100 (GPU)           & $\sim$10 \\
\bottomrule
\end{tabular}
\end{center}

\artin

\artinpart{Hardware}

\emph{CPU}: Intel Xeon Gold 6258R @ 2.4\,GHz (28 cores).

\emph{GPU}: NVIDIA H100 (80\,GB).

\artinpart{Software}

Ginkgo (high-performance numerical linear algebra library)\footnote{https://github.com/ginkgo-project/ginkgo}, OpenMP, CMake $\ge$ 3.16.

\emph{CPU}: Intel oneAPI 2025.1.

\emph{GPU}: CUDA 12.1, cuBLAS, cuSPARSE.

\artinpart{Datasets / Inputs}

7 representative sparse matrices from the SuiteSparse Matrix Collection, selected to cover a range of sparsity patterns and problem sizes. A right-hand side vector $b$ is randomly generated for each matrix. The polynomial degree is fixed at 40.

\artinpart{Installation and Deployment}

Both CPU and GPU versions use CMake as the build system. The artifacts are compiled in double precision in Release mode with \texttt{-O3}.

\emph{CPU}: Set \texttt{Ginkgo\_DIR} to the Ginkgo installation path, then \texttt{cmake -B build \&\& cmake --build build}.

\emph{GPU}: Additionally requires CUDA Toolkit $\ge$ 11.7. The CMake configuration automatically detects and links cuBLAS and cuSPARSE.

\artcomp

\emph{Workflow:}
\begin{enumerate}
  \item For each of the 7 matrices, read the matrix and generate a random RHS vector $b$.
  \item Compute the largest eigenvalue $\lambda_{\max}$ via power iteration to normalize the matrix.
  \item Build the polynomial preconditioner using coefficients derived from Chebyshev/Newton basis.
  \item Run each GMRES configuration 100 times, measuring average execution time and iteration count.
  \item The solver uses a Krylov dimension of 50, a relative residual tolerance of $10^{-5}$ (CPU) or $10^{-6}$ (GPU), and a maximum of 1000 iterations.
\end{enumerate}

\artout

Based on the measured average execution time, we can plot Figure 14 (speedup of FastMPK-preconditioned GMRES over the baseline). The program outputs each variant's name, iteration count, and execution time to \texttt{stdout}.

\arteval{1}
\artin

\artinpart{Obtaining the Artifact}

The artifact can be downloaded from the public GitHub repositories:

\begin{itemize}
  \item CPU version: \texttt{https://github.com/qleonardo/FastMPK/CPU}
  \item GPU version: \texttt{https://github.com/qleonardo/FastMPK/GPU}
\end{itemize}

Alternatively, a single archive is available at \texttt{https://doi.org/10.5281/zenodo.XXXXXXX}.

\artinpart{Software Dependencies}

\emph{CPU}: Intel oneAPI 2025.1 (with MKL) on Intel platforms; GCC $\ge$ 11.2 on AMD and ARM platforms. OpenMP is required (included with the compilers).

\emph{GPU}: CUDA Toolkit $\ge$ 11.7, cuSPARSE (included with CUDA).

\artinpart{Datasets}

Download the 110 SuiteSparse matrices (those with $\text{nnz} \ge 8\times 10^6$) from \texttt{https://sparse.tamu.edu/} in Matrix Market (.mtx) format. Place them under a \texttt{matrix/} directory structured as \texttt{matrix/<name>/<name>.mtx}.

\artinpart{Building the Artifact}

\emph{CPU (Intel)}:
\begin{verbatim}
cd FastMPK/CPU
module load intel/2025.1   # if using modules
make
\end{verbatim}

\emph{CPU (AMD / Huawei)}: Modify the \texttt{Makefile} to use \texttt{g++} instead of \texttt{icpc} and remove the MKL linking flags (or link an available BLAS). Then run \texttt{make}.

\emph{GPU}:
\begin{verbatim}
cd FastMPK/GPU
module load cuda/12.1      # H100; use cuda/11.7 for A100
make double                 # double-precision; or 'make float'
\end{verbatim}

\artinpart{Deployment}

The executable runs directly on the target machine. On CPU platforms, set thread affinity and OpenMP threads, e.g.:
\begin{verbatim}
export OMP_NUM_THREADS=48
numactl --cpunodebind=0 --membind=0 ./main \
    matrix/af_shell1/af_shell1.mtx af_shell1 5
\end{verbatim}

On GPU platforms, ensure one GPU is visible (e.g., \texttt{export CUDA\_VISIBLE\_DEVICES=0}) and run:
\begin{verbatim}
./main-double matrix/af_shell1/af_shell1.mtx af_shell1 5
\end{verbatim}

\artcomp

The experiment consists of two tasks:

$T_1$ \textbf{(Data Preparation)}: For each matrix $\mathbf{A}$, generate a random dense vector $\mathbf{x}$ of matching dimension. This is done automatically by the program on each run.

$T_2$ \textbf{(Benchmark Execution)}: For each matrix, run the executable with $K \in \{1, 3, 5, 7, 9\}$. Each run internally executes 100 iterations of each MPK variant and reports the average time. The script \texttt{run3.sh} (CPU) or \texttt{work-all-double.sh} (GPU) can be used to batch-process all matrices.

\artout

Each run appends one row to \texttt{performance\_result.csv} with the following columns:
\begin{verbatim}
matrix_name, m, nnz, CSR_GFLOPS, MKL/CUSP_GFLOPS,
FastMPK_woL_GFLOPS, FastMPK_GFLOPS
\end{verbatim}

The paper's Table IV and Figures 3, 5, 9--13 are generated by aggregating these CSV files across all matrices and $K$ values. The key metric is GFLOPS; higher values indicate better performance. FastMPK should outperform the other variants on the majority of matrices, especially for larger $K$ values and larger matrices where cache reuse is most beneficial.


\arteval{2}
\artin

\artinpart{Obtaining the Artifact}

\begin{itemize}
  \item CPU version: \texttt{https://github.com/qleonardo/FastMPK/GMRES}
  \item GPU version: \texttt{https://github.com/qleonardo/FastMPK/GMRES}
\end{itemize}

Both CPU and GPU implementations are hosted in the same repository (separate branches or subdirectories). Alternatively, use the Zenodo archive.

\artinpart{Software Dependencies}

\emph{Both}: CMake $\ge$ 3.16, Ginkgo (build from source or install via package manager: \texttt{https://github.com/ginkgo-project/ginkgo}), OpenMP.

\emph{CPU}: Intel oneAPI 2025.1 (or GCC $\ge$ 11.2 with OpenMP).

\emph{GPU}: CUDA Toolkit $\ge$ 11.7 with cuBLAS and cuSPARSE.

\artinpart{Datasets}

Download 7 representative SuiteSparse matrices (\texttt{.mtx} format). Place them under a \texttt{matrix/} directory.

\artinpart{Building the Artifact}

\emph{CPU}:
\begin{verbatim}
cd GMRES/CPU
cmake -B build \
    -DGinkgo_DIR=/path/to/ginkgo/lib/cmake/Ginkgo
cmake --build build
\end{verbatim}

\emph{GPU}:
\begin{verbatim}
cd GMRES/GPU
cmake -B build \
    -DGinkgo_DIR=/path/to/ginkgo/lib/cmake/Ginkgo \
    -DCMAKE_CUDA_ARCHITECTURES=90   # 90 for H100, 80 for A100
cmake --build build
\end{verbatim}

\artinpart{Deployment}

\emph{CPU}:
\begin{verbatim}
export OMP_NUM_THREADS=28
./build/bench matrix/af_shell1/af_shell1.mtx
\end{verbatim}

\emph{GPU}:
\begin{verbatim}
./build/bench matrix/af_shell1/af_shell1.mtx
\end{verbatim}

\artcomp

The experiment consists of two tasks:

$T_1$ \textbf{(Data Preparation)}: Read each matrix $\mathbf{A}$ from \texttt{.mtx} format via Ginkgo's built-in reader. Compute $\lambda_{\max}$ via power iteration. Generate a random RHS vector $\mathbf{b}$.

$T_2$ \textbf{(Solver Execution)}: For each matrix, run the three GMRES configurations (Ginkgo-native, CSR-MPK, FastMPK) 100 times each. Each run measures wall-clock time for the full solve (including preconditioner setup). The polynomial degree is 40, Krylov dimension is 50, and relative residual tolerance is $10^{-5}$ (CPU) or $10^{-6}$ (GPU).

\artout

The program outputs to \texttt{stdout} for each variant:
\begin{verbatim}
<variant_name>, <iterations>, <time_seconds>
\end{verbatim}

Figure 14 is generated by computing the speedup of FastMPK-preconditioned GMRES over the Ginkgo-native baseline: $\text{Speedup} = T_{\text{Ginkgo}} \;/\; T_{\text{FastMPK}}$. The FastMPK variant should consistently achieve higher speedups (lower execution times) across the test matrices, demonstrating the practical benefit of cache-reuse-based polynomial preconditioning in real iterative solvers.
